üìä Database Schema‚Äã

Core Models‚Äã

User Model‚Äã

model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  mode      UserMode @default(STUDENT)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // Relationships
  blocks    Block[]
  folders   Folder[]
  pointsUpdates PointsUpdate[]

  @@map("users")
}

Block Model (Content)‚Äã

model Block {
  id        String   @id @default(cuid())
  title     String
  content   String
  authorId  String
  folderId  String?
  published Boolean  @default(false)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // Relationships
  author    User     @relation(fields: [authorId], references: [id])
  folder    Folder?  @relation(fields: [folderId], references: [id])
  questions Question[]
  quizzes   Quiz[]
  topics    Topic[]
  fillInTheBlanks FillInTheBlank[]
  pointsUpdates PointsUpdate[]

  @@map("blocks")
}

Folder Model (Organization)‚Äã

model Folder {
  id          String   @id @default(cuid())
  name        String
  description String?
  authorId    String
  parentId    String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  // Relationships
  author      User     @relation(fields: [authorId], references: [id])
  parent      Folder?  @relation("FolderHierarchy", fields: [parentId], references: [id])
  children    Folder[] @relation("FolderHierarchy")
  blocks      Block[]

  @@map("folders")
}

model Quiz {

üîß Prisma Operations‚Äã

Schema Management‚Äã

Generate Prisma Client‚Äã

bunx prisma generate

When to use:

  • After schema changes
  • After pulling database changes
  • When TypeScript shows Prisma errors
  • Before running the application

Create Migration‚Äã

bunx prisma migrate dev --name descriptive_name

What it does:

  • Detects schema changes
  • Creates migration file
  • Applies migration to database
  • Regenerates Prisma client
  • Seeds database (if configured)

Apply Migrations‚Äã

bunx prisma migrate deploy

For production:

  • Applies pending migrations
  • No schema generation
  • Safe for production use

Reset Database‚Äã

bunx prisma migrate reset

‚ö†Ô∏è Warning: Destructive operation!

  • Drops all tables
  • Recreates schema
  • Applies all migrations
  • Runs seed script

Database Operations‚Äã

Pull Schema Changes‚Äã

bunx prisma db pull

When to use:

  • Database changed externally
  • Working with existing database
  • Syncing with production schema

Push Schema Changes‚Äã

bunx prisma db push

Development only:

  • Pushes schema to database
  • No migration files created
  • Useful for prototyping

Validate Schema‚Äã

bunx prisma validate

What it checks:

  • Schema syntax
  • Model relationships
  • Field types
  • Constraints

Database GUI‚Äã

Prisma Studio‚Äã

bunx prisma studio

Features:

  • Visual database browser
  • Edit data directly
  • View relationships
  • Export data
  • Runs on http://localhost:5555

üìù Migration Workflow‚Äã

Creating Schema Changes‚Äã

  1. Modify Schema

    // prisma/schema.prisma
    model User {
      id        String   @id @default(cuid())
      email     String   @unique
      name      String?
      // Add new field
      avatar    String?
      createdAt DateTime @default(now())
      updatedAt DateTime @updatedAt
    }

  2. Create Migration

    bunx prisma migrate dev --name add_user_avatar

  3. Verify Changes

    bunx prisma studio

Migration Best Practices‚Äã

Naming Conventions‚Äã

# Good names
bunx prisma migrate dev --name add_user_avatar
bunx prisma migrate dev --name update_quiz_time_limit
bunx prisma migrate dev --name create_topic_index

# Avoid generic names
bunx prisma migrate dev --name migration
bunx prisma migrate dev --name update

Testing Migrations‚Äã

# Test on development database first
bunx prisma migrate dev

# Verify data integrity
bunx prisma studio

# Check application functionality
bun coolDev

Production Migrations‚Äã

# Create migration file only
bunx prisma migrate dev --create-only

# Review migration file
# Edit if needed

# Apply to production
bunx prisma migrate deploy

üîç Database Queries‚Äã

Using Prisma Client‚Äã

Basic Queries‚Äã

import { prisma } from '@/lib/generated/prisma';

// Find single record
const user = await prisma.user.findUnique({
  where: { id: 'user-123' }
});

// Find many records
const users = await prisma.user.findMany({
  where: { mode: 'STUDENT' }
});

// Create record
const newUser = await prisma.user.create({
  data: {
    email: 'user@example.com',
    name: 'John Doe',
    mode: 'STUDENT'
  }
});

// Update record
const updatedUser = await prisma.user.update({
  where: { id: 'user-123' },
  data: { name: 'Jane Doe' }
});

// Delete record
const deletedUser = await prisma.user.delete({
  where: { id: 'user-123' }
});

Relationship Queries‚Äã

// Include related data
const userWithBlocks = await prisma.user.findUnique({
  where: { id: 'user-123' },
  include: {
    blocks: true,
    folders: true
  }
});

// Nested includes
const blockWithAuthor = await prisma.block.findUnique({
  where: { id: 'block-123' },
  include: {
    author: {
      select: {
        id: true,
        name: true,
        email: true
      }
    },
    topics: true,
    questions: true
  }
});

Advanced Queries‚Äã

// Complex where conditions
const publishedBlocks = await prisma.block.findMany({
  where: {
    published: true,
    author: {
      mode: 'TEACHER'
    },
    createdAt: {
      gte: new Date('2024-01-01')
    }
  },
  include: {
    author: true,
    topics: true
  },
  orderBy: {
    createdAt: 'desc'
  },
  take: 10
});

// Aggregation
const userStats = await prisma.user.aggregate({
  where: { mode: 'STUDENT' },
  _count: {
    blocks: true,
    folders: true
  },
  _sum: {
    pointsUpdates: {
      points: true
    }
  }
});

Indexing Strategy‚Äã

Primary Indexes‚Äã

model User {
  id    String @id @default(cuid())
  email String @unique  // Automatic index
  // ...
}

Custom Indexes‚Äã

model Block {
  id        String   @id @default(cuid())
  title     String
  authorId  String
  published Boolean  @default(false)
  createdAt DateTime @default(now())

  // Custom indexes
  @@index([authorId])
  @@index([published, createdAt])
  @@index([title])
}

Composite Indexes‚Äã

model PointsUpdate {
  id       String   @id @default(cuid())
  points   Int
  blockId  String?
  userId   String?
  createdAt DateTime @default(now())

  // Composite index for efficient queries
  @@index([userId, createdAt])
  @@index([blockId, createdAt])
}

Query Optimization‚Äã

Select Only Needed Fields‚Äã

// Efficient - only select needed fields
const userNames = await prisma.user.findMany({
  select: {
    id: true,
    name: true
  }
});

// Avoid selecting all fields unless needed
const allUserData = await prisma.user.findMany(); // Less efficient

Use Pagination‚Äã

// Paginated queries
const blocks = await prisma.block.findMany({
  take: 20,
  skip: 40, // Page 3 (20 * 2)
  orderBy: {
    createdAt: 'desc'
  }
});

Batch Operations‚Äã

// Batch create
const users = await prisma.user.createMany({
  data: [
    { email: 'user1@example.com', name: 'User 1' },
    { email: 'user2@example.com', name: 'User 2' },
    { email: 'user3@example.com', name: 'User 3' }
  ]
});

// Batch update
const updatedBlocks = await prisma.block.updateMany({
  where: { published: false },
  data: { published: true }
});

Constraints‚Äã

Unique Constraints‚Äã

model User {
  id    String @id @default(cuid())
  email String @unique  // Ensures unique emails
  // ...
}

Required Fields‚Äã

model Block {
  id       String @id @default(cuid())
  title    String  // Required field
  content  String  // Required field
  authorId String  // Required field
  // ...
}

Foreign Key Constraints‚Äã

model Block {
  id       String @id @default(cuid())
  authorId String
  folderId String?

  // Foreign key relationships
  author User @relation(fields: [authorId], references: [id])
  folder Folder? @relation(fields: [folderId], references: [id])
}

Validation‚Äã

Field Validation‚Äã

model User {
  id    String @id @default(cuid())
  email String @unique @db.VarChar(255)
  name  String? @db.VarChar(100)
  // ...
}

Custom Validation‚Äã

// In your application code
const createUser = async (data: CreateUserData) => {
  // Validate email format
  if (!data.email.includes('@')) {
    throw new Error('Invalid email format');
  }

  // Validate name length
  if (data.name && data.name.length > 100) {
    throw new Error('Name too long');
  }

  return await prisma.user.create({ data });
};
